Capillary Solution Agent

An AI assistant that helps the Professional Services team evaluate client requests in minutes instead of hours — by searching real data, not guessing.

● Production Anthropic Claude Multi-user Auth LangGraph Orchestration MongoDB Atlas

What it does

Think of it as a knowledgeable colleague who has read every Confluence page and Jira ticket — and can give you a structured answer in under a minute.

  • Submit a Jira ticket ID, describe a client requirement, or upload a document (PDF, images, text)
  • The agent searches Jira, Confluence, Capillary product docs, and docs.capillarytech.com for real precedents
  • Returns a structured assessment: Problem → Verdict → Approach → Complexity → References → Open Questions
  • When a request is too complex for routine evaluation, it flags for SA escalation with a pre-filled summary

🔵 Standard path

The research node calls tools (Jira, Confluence, Kapa, web search), gets results, and loops back through compactIfNeeded which checks if the context window is getting too large and summarises older messages if needed. When the model is done researching, it synthesises the final answer and validate checks quality.

🟣 Document path

For SDDs and gap analyses. researchFanOut searches all sources simultaneously, sectionWriter writes each section in parallel using the right model (Haiku for simple, Sonnet for complex), and skillValidate checks the output against the skill's validation rules.

Two AI models, used wisely

Not every task needs the most powerful (and expensive) AI. The agent uses a fast, affordable model for quick tasks and a powerful reasoning model only when the task genuinely requires it — like using a calculator for arithmetic and a mathematician for proofs.

Sonnet 4 Reasoning Complex analysis, multi-source synthesis, architectural decisions — the "thinking" model
Haiku 4.5 Fast tasks Classification, summarisation, simple sections, skill selection — ~50× cheaper, runs in parallel

Specialist skills

The agent can activate specialist capabilities depending on what you ask for — like calling in a subject matter expert.

🎯 CR Evaluator

Feasibility rubric loaded automatically for CR, BRD, and issue requests. Knows what's OOTB, what needs configuration, and what requires custom development in Capillary. Omitted for general queries to save tokens.

📄 SDD Writer

Generates Solution Design Documents with parallel research and section-by-section writing. Delivered as a downloadable file.

📊 Gap Analyzer

Compares a BRD against Capillary capabilities. Scores each domain, identifies gaps, and suggests resolution paths.

🖼️ Diagram Generator

Creates visual architecture diagrams as Excalidraw JSON for solution architecture visualisation.

Adding a new skill means creating a folder with instructions — no code changes needed. Skills declare their own execution mode, research needs, and validation rules via a manifest file.

Engineering optimizations

Under the hood, several optimizations keep the agent fast and cost-efficient.

⚡
Parallel pre-flight — Request classification and skill detection run simultaneously before the main loop starts, saving 300–500ms on every request.
🎯
Conditional skill loading — Only relevant skills and reference files are loaded based on what you're asking about, keeping the AI's context focused.
💰
Model dispatch policy — Every operation is routed to the cheapest model that can handle it. The expensive model is used only for complex reasoning.
🗜️
Context compaction — When conversations get long, older messages are automatically summarised by Haiku to keep the context window lean. The full history is always preserved in the database.
🚪
Off-topic gate — A fast Haiku classifier intercepts irrelevant requests before they reach the main loop, saving Sonnet tokens and keeping the agent focused.
🔧
Intent-based tool loading — Only tools relevant to the user's question are included in each API call, reducing noise and token overhead.
🔀
Parallel research agents — Domain-specific Haiku sub-agents research Jira, Confluence, and docs simultaneously instead of sequential Sonnet calls. 3-5× faster, 60-70% cheaper.

Observability with LangSmith

When enabled, every request is fully traceable — like having a flight recorder for AI conversations.

LangSmith is a monitoring platform that records each step the agent takes: what it classified, which skills it loaded, what tools it called, and how it synthesised the answer. This helps the team debug issues, monitor response quality, and track costs per operation.

It's optional — when the LANGCHAIN_TRACING_V2 environment variable isn't set, tracing is silently disabled with zero performance impact.

Database & persistence

All data is stored in MongoDB Atlas — a cloud-hosted database that provides reliability, scalability, and zero-downtime deployments.

💬 Conversations

Every message is persisted atomically. Conversations survive server restarts and are scoped per user — no one can see another user's chats.

👤 User accounts

Full user lifecycle: admin-created accounts with one-time or permanent passwords, SSO via Google/Microsoft, password policies, and account lockout after failed attempts.

🏢 Client personas

Client-specific context (modules, known issues, recent interactions) stored in the database and automatically injected when the agent detects a known client.

📋 Audit trail

Every security event — logins, password changes, user creation, account lockouts — is logged with timestamps and actor identification.

The system uses a store adapter pattern — all code talks to a generic interface, and the backend (MongoDB or JSON files) is swapped via a single environment variable. This made the migration from flat files to MongoDB a zero-change operation for all application code.

User management

Admins have full control over user accounts through a dedicated management interface.

👥
Admin panel — Create, edit, and delete user accounts. Assign roles (admin/user) and set password types (one-time or permanent).
🔑
Password policies — Minimum 8 characters with uppercase, lowercase, digit, and special character requirements. Enforced on creation, change, and reset.
🔒
Account lockout — After 5 consecutive failed login attempts, the account is locked for 15 minutes. Admins can reset lockouts manually.
🔄
Forced password change — Users created with one-time passwords must change their password on first login before accessing the application.
🌐
SSO integration — Google and Microsoft OAuth with domain restriction. SSO users are auto-provisioned on first login.

Agent planning

For complex requests, the agent automatically creates a structured plan and tracks progress in real-time.

📋 Auto-planning

The agent automatically creates a plan when the request involves multiple research steps, sequential actions, or multi-part analysis. Simple lookups skip planning entirely.

📊 Live progress

Plan cards appear in the chat showing each step with status icons — pending ○, in progress ●, completed ✓, skipped ⊘. Steps update in real-time as the agent works.

💾 Persistent plans

Plans are saved alongside conversations in MongoDB. Reload the page or come back later — the plan state is preserved exactly as you left it.

▶ Continue Plan

If a plan has incomplete steps, a "Continue Plan" button lets you resume where the agent left off — enabling long-running research or document writing across multiple turns.

Parallel research agents

Instead of searching sources one at a time, the agent dispatches specialist sub-agents that research in parallel — then synthesises everything in a single pass.

🔀
Parallel dispatch — Jira, Confluence, Kapa docs, and web search agents run simultaneously, each powered by the fast Haiku model. Research that used to take 15-20 seconds now completes in 3-5 seconds.
🧠
Query reformulation — Follow-up questions like "any other way?" are automatically reformulated into specific search terms using conversation context, so research agents find relevant results.
📝
Structured summaries — Each research agent returns a concise structured summary (~500 tokens) instead of raw tool output. The synthesis model sees clean, focused context.
🛡️
Graceful fallback — If all parallel agents fail, the system automatically falls back to the sequential research loop. The user always gets a response.
